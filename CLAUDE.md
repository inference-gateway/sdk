# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

All commands run via [Task](https://taskfile.dev):

- `task test` — run the full test suite (`go test -v ./...`)
- `task lint` — run `golangci-lint`
- `task generate` — regenerate `generated_types.go` from `openapi.yaml` via `oapi-codegen`
- `task oas-download` — refresh `openapi.yaml` from the upstream `inference-gateway/schemas` repo
- `task tidy` — `go mod tidy` across all modules
- `task docs` — serve godoc on `:6060`

Run a single test: `go test -v -run TestName ./...` (e.g. `go test -v -run TestGenerateContent ./...`).

A [Flox](https://flox.dev) environment under `.flox/` pins the Go toolchain and `golangci-lint` to the CI versions. `flox activate` is the recommended way to get a matching shell; otherwise Go 1.26+ is required.

## Architecture

This is a thin Go client for the [Inference Gateway](https://github.com/inference-gateway), a unified API that fronts multiple LLM providers (OpenAI, Anthropic, Groq, Ollama, etc.).

**OpenAPI-driven types.** The SDK is contract-bound to `openapi.yaml`. `generated_types.go` is fully generated by `oapi-codegen` — never edit it by hand. To add or change a request/response shape, update the upstream schema, then `task oas-download && task generate`. Hand-written types and helpers (`ClientOptions`, `RetryConfig`, `MiddlewareOptions`, `NewMessageContent`, `NewTextMessage`, `NewImageMessage`, etc.) live in `types.go`.

**Client surface.** `sdk.go` defines the `Client` interface and its sole implementation `clientImpl`. The fluent `WithX` methods (`WithAuthToken`, `WithTools`, `WithOptions`, `WithHeaders`, `WithHeader`, `WithMiddlewareOptions`) all return `*clientImpl` so calls can be chained before a terminal operation like `GenerateContent` / `GenerateContentStream` / `ListModels` / `ListProviderModels` / `ListTools` / `HealthCheck`.

**Transport.** Built on `go-resty`. All requests go through `executeWithRetry`, which implements exponential backoff with jitter, honors the `Retry-After` header on 429s, and respects context cancellation. Retry-eligible status codes default to 408/429/500/502/503/504 and are overridable via `RetryConfig.RetryableStatusCodes`. `isRetryableError` also retries common transient network errors (dial/read failures, DNS errors, EOF, deadline exceeded).

**Streaming.** `GenerateContentStream` returns a `<-chan SSEvent` consumed by the caller. Event types — `ContentDelta`, `StreamEnd`, `MessageError` — are the gateway's SSE envelope; the payload inside `Data` is JSON that the caller unmarshals into `CreateChatCompletionStreamResponse`.

**Vision / multimodal.** `Message.Content` is a oneOf (`string` | `[]ContentPart`) generated by `oapi-codegen` as a union with `FromMessageContent0` / `FromMessageContent1`. The helper functions in `types.go` (`NewMessageContent`, `NewTextContentPart`, `NewImageContentPart`, `NewImageMessage`) hide this and are the intended API surface.

**Middleware bypass.** The gateway runs middleware layers (notably MCP). `MiddlewareOptions{SkipMCP, DirectProvider}` translate to `X-MCP-Bypass` / `X-Direct-Provider` headers on the wire — the same effect can be achieved by setting headers directly.

## Conventions

- TDD: write/extend tests in `sdk_test.go` or `vision_test.go` before implementing. Table-driven tests are the default style — follow the patterns already in those files.
- Run `go fmt ./...` before committing.
- Conventional Commits are required (`feat:`, `fix:`, `refactor:`, `docs:`, `style:`, `test:`, `chore:`, `ci:`, `perf:`). Release versioning and changelog generation are driven by commit messages via `semantic-release` (`.releaserc.yaml`).
- All changes ship via Pull Requests; merging to `main` triggers CI, releases are run manually from Actions.
